## ⚠️ IMPORTANT: Strategic Decisions After Client Review (Feb 17, 2026\)

Following detailed client discussion, several **critical strategic decisions** have been made that significantly impact implementation:

### 1\. 🏗️ **MODULAR ARCHITECTURE REQUIRED**

The system must be built with **modular architecture** from the start to enable:

- Per-module subscription pricing  
- Customers pay only for features they use  
- Ability to enable/disable modules per company

**Module Structure**:

- **Base Module**: Drivers, vehicles, clients, companies (always included)  
- **Planning Module**: Ride planning, scheduling, hours (optional subscription)  
- **Finance Module**: Payroll, invoicing, reports (optional subscription)  
- **HR Module**: Contracts, HR letters, employee management (optional subscription)

.

### 2\. 📦 **DROPBOX/GOOGLE DRIVE INTEGRATION** (Major Simplification)

Instead of building custom file storage system:

- ✅ Integrate with Dropbox API or Google Drive API  
- ✅ Documents generated by app → Auto-upload to Dropbox  
- ✅ Admin/driver uploads → Save to Dropbox  
- ✅ Document viewing → Download links from Dropbox  
- ✅ 7-year retention → Handled by Dropbox  
- ✅ Customer pays Dropbox costs directly

**Affected Requirements**: R26, R28, R31 (significant time savings: \~90-110 hours)

### 3\. 📊 **SIMPLIFIED REPORTING FOCUS**

Several requirements are primarily **report generation** from existing data, not new complex systems:

- R32: Admin override form for starting balances (not salary engine)  
- R34, R35: Report generation from existing calculations  
- R36: Aggregation report of invoices from R37

**Impact**: Significant complexity reductions across multiple requirements.

## R23: Potential Driver Management (Company Linking)

### Requirement Summary

Customer admins should be able to link a **potential driver** (prospect/candidate) to a specific company in the system before they become an active driver.

**Business Context**:

- Recruitment pipeline management  
- Track driver prospects/candidates before hiring  
- Associate candidates with company early in process  
- Prevent duplicate data entry when converting prospect → driver

### Current State

#### What exists now:

✅ **Driver entity**: Full driver records with employment details ✅ **Company entity**: Company records that drivers belong to ✅ **Customer admin role**: Can manage their company's data

#### Current gaps:

- ❌ **No "Potential Driver" or "Prospect" entity**  
- ❌ **Cannot track candidates before they become drivers**  
- ❌ **No recruitment pipeline/status tracking**  
- ❌ **Customer admin cannot add prospects to their company**  
- ❌ **Must create full driver record immediately (even for prospects)**

### Gap Analysis

**What needs to be added:**

1. **PotentialDrivers table**:  
     
   - Basic info: Name, email, phone  
   - CompanyId (link to company)  
   - Status (New, Contacted, Interviewing, Offer Made, Accepted, Rejected, Converted to Driver)  
   - Notes/comments  
   - Source (referral, job board, etc.)  
   - Expected start date  
   - ConvertedToDriverId (when prospect becomes driver)

   

2. **Potential Drivers management page**:

3. **Authorization**:

## R24: Multiple Contract Types (CAO, ZZP, Inleen, Brief Loonschaal)

### Requirement Summary

Support **4 different contract types** for drivers with type-specific data fields and payment calculations:

1. **CAO** (Collective Labor Agreement \- current system)  
2. **ZZP** (Zelfstandige Zonder Personeel \- Self-employed/freelance)  
3. **Inleen** (Secondment \- between companies)  
4. **Brief Loonschaal** (Simple wage scale \- different from CAO)

### Current State

#### What exists now:

✅ **CAO contract support**:

- `EmployeeContract` entity with CAO-specific fields  
- PayScale, PayScaleStep, HourlyWage100Percent  
- CAO vacation days calculation  
- CAO payment calculation in ReportCalculationService

✅ **Contract PDF generation**:

- `DriverContractPdfBuilder` service  
- Generates CAO-style contracts

✅ **Invoice/Timesheet generation**:

- `DriverInvoicePdfBuilder`  
- `DriverTimesheetPdfGenerator`

#### Current gaps:

- ❌ **No ContractType field** \- all contracts assumed to be CAO  
- ❌ **No ZZP-specific fields** (BTW number, KVK, hourly rate excl. BTW)  
- ❌ **No Inleen-specific fields** (lending company, borrowing company, dates, rate)  
- ❌ **No Brief-specific fields** (monthly salary, grade/level)  
- ❌ **Payment calculation only supports CAO logic**  
- ❌ **Contract PDF generation only for CAO format**  
- ❌ **Invoice generation doesn't differentiate contract types**

### Gap Analysis

**What needs to be added:**

##### 2\. Backend Services

**Extend ReportCalculationService**:

public class ReportCalculationService

{

    public DriverTimesheetReport CalculateWeeklyReport(Guid driverId, int year, int weekNumber)

    {

        var driver \= GetDriver(driverId);

        var contract \= driver.Contract;

        

        // Branch based on contract type

        switch (contract.ContractType)

        {

            case ContractType.CAO:

                return CalculateCAOReport(driver, year, weekNumber);

            case ContractType.ZZP:

                return CalculateZZPReport(driver, year, weekNumber);

            case ContractType.Inleen:

                return CalculateInleenReport(driver, year, weekNumber);

            case ContractType.BriefLoonschaal:

                return CalculateBriefReport(driver, year, weekNumber);

            default:

                throw new NotSupportedException($"Contract type {contract.ContractType} not supported");

        }

    }

    

    private DriverTimesheetReport CalculateZZPReport(Driver driver, int year, int weekNumber)

    {

        // Calculate hours \* rate

        var hours \= GetHoursWorked(driver, year, weekNumber);

        var rate \= driver.Contract.HourlyRateExclBtw;

        var subtotal \= hours \* rate;

        

        // Add BTW

        var btwAmount \= subtotal \* (driver.Contract.BtwPercentage / 100);

        var total \= subtotal \+ btwAmount;

        

        // No vacation, no TvT for ZZP

        return new DriverTimesheetReport

        {

            TotalHours \= hours,

            SubtotalExclBtw \= subtotal,

            BtwAmount \= btwAmount,

            TotalInclBtw \= total,

            // No vacation/TvT fields

        };

    }

}

## R25: Comprehensive Contract System (All Employee Types \+ Contract Variations)

### Requirement Summary

Expand contract management to **ALL employees** (not just drivers) with support for **10+ contract types and variations**:

**Employee Types**: Drivers, Planners, Administrators, Managers, Accountants

**Contract Types**:

1. **CAO** (Collective Labor Agreement)  
2. **ZZP** (Self-employed)  
3. **Inleen** (Secondment)  
4. **Brief Loonschaal** (Simple wage scale)  
5. **OVO** (Opleidings- en Vormingsovereenkomst \- Training agreement)  
6. **Uitzend** (Temporary/Agency work)  
7. **Raam** (Framework agreement)  
8. **Bemiddeling** (Intermediary/Placement)  
9. **Deeltijd** (Part-time)

**Contract Variations**:

- Fixed-term (Bepaald) vs. Permanent (Onbepaald)  
- External vs. Internal  
- Full-time vs. Part-time

### Current State

#### What exists now:

✅ **Driver-centric system**:

- `Driver` entity with `EmployeeContract`  
- CAO contracts for drivers

✅ **Contact persons** (non-driver staff):

- `ContactPerson` entity for customer admins, planners  
- No contracts tracked

✅ **Users**:

- `ApplicationUser` (ASP.NET Identity)  
- Roles: globalAdmin, customerAdmin, employer, planner, driver, etc.

#### Current gaps:

- ❌ **No unified Employee entity** \- drivers and staff are separate  
- ❌ **No contracts for non-driver staff** (planners, admins, accountants)  
- ❌ **No support for 10+ contract types** (only CAO exists)  
- ❌ **No contract variations** (fixed-term vs permanent, external vs internal)  
- ❌ **No employee role tracking** (linked to employment)  
- ❌ **Payment/payroll only for drivers** (not for office staff)

### Gap Analysis

**This is a MAJOR architectural change requiring:**

1. **New unified Employee entity**:  
     
   - Replaces separate Driver and ContactPerson entities  
   - Or: Keep Driver but link to Employee (recommended)  
   - Fields: EmployeeId, Role (Driver, Planner, Admin, etc.), EmploymentType

   

2. **Expanded contract types** (beyond R24's 4 types):  
     
   - Add 6 more contract types: OVO, Uitzend, Raam, Bemiddeling, Deeltijd  
   - Each type with specific fields and rules

   

3. **Contract variations**:  
     
   - `ContractDuration`: FixedTerm, Permanent  
   - `EmploymentType`: External, Internal  
   - `WorkingHours`: FullTime, PartTime (with % or hours)

   

4. **Employee management**:  
     
   - CRUD for all employee types  
   - Role assignment  
   - Contract assignment  
   - Payroll for all (not just drivers)

   

5. **Payment/reporting**:  
     
   - Extend payroll to office staff  
   - Different payment rules per role and contract type

### Proposed Architecture

**Option 1: Unified Employee Table** (Recommended):

Employee (replaces Driver \+ ContactPerson)

├── Id

├── AspNetUserId

├── EmployeeRole (Driver, Planner, Admin, Manager, Accountant)

├── CompanyId

├── FirstName, LastName, Email, Phone

├── EmployeeContractId → EmployeeContract

└── (role-specific details in separate tables if needed)

**Option 2: Keep Driver, Add Employee** (Alternative):

Employee (new, for office staff)

**Dropbox Integration Service**:

public class DropboxStorageService

{

    private readonly DropboxClient \_dropbox;

    

    public async Task\<DropboxUploadResult\> UploadSupplierDocument(

        Guid supplierId, 

        string supplierName,

        string documentType,

        IFormFile file)

    {

        // Create folder path: /Suppliers/{SupplierName}/{DocumentType}/

        var folderPath \= $"/Suppliers/{supplierName}/{documentType}";

        var dropboxPath \= $"{folderPath}/{file.FileName}";

        

        // Upload to Dropbox

        using var stream \= file.OpenReadStream();

        var uploadResult \= await \_dropbox.Files.UploadAsync(

            dropboxPath,

            WriteMode.Overwrite.Instance,

            body: stream

        );

        

        // Generate shareable link

        var linkResult \= await \_dropbox.Sharing.CreateSharedLinkWithSettingsAsync(

            uploadResult.PathDisplay,

            new SharedLinkSettings { RequestedVisibility \= RequestedVisibility.Public }

        );

        

        return new DropboxUploadResult

        {

            DropboxPath \= uploadResult.PathDisplay,

            DropboxFileId \= uploadResult.Id,

            ShareLink \= linkResult.Url

        };

    }

    

    public async Task\<byte\[\]\> DownloadDocument(string dropboxPath)

    {

        var response \= await \_dropbox.Files.DownloadAsync(dropboxPath);

        return await response.GetContentAsByteArrayAsync();

    }

    

    public async Task DeleteDocument(string dropboxPath)

    {

        await \_dropbox.Files.DeleteV2Async(dropboxPath);

    }

}

### Requirement Summary

Generate formal **payroll and tax reports** in standard Dutch formats:

1. **UVS** (Uniform Wage Statement format) \- Standardized format for social security  
2. **Payroll Statements** (Loonstroken) \- Monthly/period salary statements

**AnnualStatementCalculationService** (new):

public class AnnualStatementCalculationService

{

    public async Task\<AnnualStatementData\> CalculateAnnualTotals(Guid driverId, int year)

    {

        var driver \= await \_db.Drivers

            .Include(d \=\> d.User)

            .Include(d \=\> d.Contract)

            .FirstOrDefaultAsync(d \=\> d.Id \== driverId);

        

        var startDate \= new DateTime(year, 1, 1);

        var endDate \= new DateTime(year, 12, 31);

        

        // Get all weekly/period reports for the year

        var periods \= GetPeriodsForYear(year);

        

        decimal totalGross \= 0;

        decimal totalAllowances \= 0;

        double totalHours \= 0;

        double totalHours100 \= 0, totalHours130 \= 0, totalHours150 \= 0, totalHours200 \= 0;

        

        foreach (var period in periods)

        {

            var report \= await \_reportCalcService.CalculateWeeklyReport(driverId, year, period);

            

            totalGross \+= report.GrandTotal.TotalWage;

            totalAllowances \+= report.GrandTotal.TotalAllowances;

            totalHours \+= report.GrandTotal.TotalHours;

            totalHours100 \+= report.HoursSummary.Hours100;

            totalHours130 \+= report.HoursSummary.Hours130;

            totalHours150 \+= report.HoursSummary.Hours150;

            totalHours200 \+= report.HoursSummary.Hours200;

        }

        

        // Vacation calculation

        var vacationEntitled \= await \_vacationCalc.GetAnnualEntitlement(driverId, year);

        var vacationTaken \= await \_vacationCalc.GetVacationTaken(driverId, year);

        

        // Vacation allowance (8% of gross in NL)

        decimal vacationAllowance \= totalGross \* 0.08m;

        

        return new AnnualStatementData

        {

            Year \= year,

            TotalGrossWage \= totalGross,

            TotalAllowances \= totalAllowances,

            TotalHoursWorked \= totalHours,

            TotalHours100 \= totalHours100,

            TotalHours130 \= totalHours130,

            TotalHours150 \= totalHours150,

            TotalHours200 \= totalHours200,

            VacationDaysEntitled \= vacationEntitled / 8, // Convert hours to days

            VacationDaysTaken \= vacationTaken / 8,

            VacationDaysRemaining \= (vacationEntitled \- vacationTaken) / 8,

            VacationAllowancePaid \= vacationAllowance,

            TotalNetPayment \= totalGross \+ totalAllowances \- 0, // No deductions in Phase 1

            EmploymentStartDate \= driver.Contract.StartDate,

            EmploymentEndDate \= driver.Contract.EndDate,

            MonthsEmployed \= CalculateMonthsEmployed(driver.Contract.StartDate, driver.Contract.EndDate, year)

        };

    }

}

## R35: Annual Statement Workflows (Leaving vs. Active Employees) \- FULL ANALYSIS

### Requirement Summary

Implement **distinct workflows** for generating annual statements (jaaropgave) based on employee employment status:

1. **Employees leaving employment** \- Generate jaaropgave immediately upon termination/resignation (mid-year)  
2. **Employees in employment** \- Generate jaaropgave at year-end (January for previous year)

**Business Context**:

- **Legal requirement**: Employees leaving mid-year need jaaropgave within 4 weeks of departure (for tax filing & new employment)  
- **Year-end batch**: Active employees receive jaaropgave in January for the previous tax year  
- **Pro-rated calculations**: Leaving employees' jaaropgave covers only months worked (partial year)  
- **Administrative burden**: Must not forget to generate jaaropgave for departing employees  
- **Compliance**: Missing or delayed jaaropgave can result in fines from tax authorities

### Current State

✅ **Annual statement generation**: Can generate jaaropgave for any driver ✅ **Employment dates tracking**: EmployeeContract has start/end dates ✅ **Status tracking**: EmployeeContractStatus enum (Active, Terminated, etc.) ✅ **Timesheet calculations**: ReportCalculationService can calculate for any date range

#### Current gaps:

- ❌ **No automatic jaaropgave trigger when employee leaves**  
- ❌ **No distinction in workflow** (leaving vs. active)  
- ❌ **No year-end batch generation**  
- ❌ **No "pending jaaropgave" tracking**  
- ❌ **No notification to departing employees**  
- ❌ **No deadline tracking** (4-week compliance deadline)  
- ❌ **No overdue alerts**

public async Task\<AnnualStatementData\> CalculateForDepartingEmployee(

    Guid driverId, 

    DateTime terminationDate)

{

    var driver \= await \_db.Drivers.FindAsync(driverId);

    var contract \= driver.Contract;

    

    // Determine calculation period (partial year)

    var year \= terminationDate.Year;

    var startDate \= new DateTime(year, 1, 1);

    

    // If hired during this year, start from hire date

    if (contract.StartDate.Year \== year && contract.StartDate \> startDate)

    {

        startDate \= contract.StartDate;

    }

    

    var endDate \= terminationDate;

    

    // Calculate months employed (for pro-rating)

    int monthsEmployed \= ((endDate.Year \- startDate.Year) \* 12\) \+ endDate.Month \- startDate.Month \+ 1;

    

    // Calculate totals for this period only

    var totals \= await CalculateTotalsForPeriod(driverId, startDate, endDate);

    

    // Pro-rate vacation entitlement

    double annualVacation \= await \_vacationCalc.GetAnnualEntitlement(driverId, year);

    double proratedVacation \= (annualVacation / 12.0) \* monthsEmployed;

    

    return new AnnualStatementData

    {

        Year \= year,

        IsPartialYear \= true,

        CalculationStartDate \= startDate,

        CalculationEndDate \= endDate,

        MonthsEmployed \= monthsEmployed,

        TotalGrossWage \= totals.Gross,

        // ... other fields

        VacationDaysEntitled \= (int)(proratedVacation / 8), // Pro-rated

        EmploymentEndDate \= terminationDate

    };

}

public class AnnualStatementWorkflowService

{

    public async Task\<List\<PendingJaaropgave\>\> GetPendingDepartures()

    {

        // Find terminated employees without jaaropgave

        var recentlyTerminated \= await \_db.Drivers

            .Include(d \=\> d.User)

            .Include(d \=\> d.Contract)

            .Where(d \=\> d.Contract.Status \== EmployeeContractStatus.Terminated)

            .Where(d \=\> d.Contract.EndDate \>= DateTime.Today.AddMonths(-6)) // Last 6 months

            .ToListAsync();

        

        var pending \= new List\<PendingJaaropgave\>();

        

        foreach (var driver in recentlyTerminated)

        {

            var year \= driver.Contract.EndDate.Value.Year;

            

            // Check if jaaropgave exists for this year

            var hasStatement \= await \_db.AnnualStatements

                .AnyAsync(s \=\> s.DriverId \== driver.Id && s.Year \== year);

            

            if (\!hasStatement)

            {

                var daysSince \= (DateTime.Today \- driver.Contract.EndDate.Value).Days;

                

                pending.Add(new PendingJaaropgave

                {

                    DriverId \= driver.Id,

                    DriverName \= driver.User.FullName,

                    TerminationDate \= driver.Contract.EndDate.Value,

                    DaysSinceTermination \= daysSince,

                    IsOverdue \= daysSince \> 28

                });

            }

        }

        

        return pending.OrderByDescending(p \=\> p.DaysSinceTermination).ToList();

    }

}

public class FinancialSummaryService

{

    public async Task\<FinancialSummaryReport\> GenerateReport(int year, int? month \= null)

    {

        // Get customer invoices from R37 (Debtors/Revenue)

        var customerInvoices \= await \_db.CustomerInvoices

            .Where(i \=\> i.Year \== year && (\!month.HasValue || i.Month \== month))

            .Include(i \=\> i.Client)

            .ToListAsync();

        

        // Get self-billing invoices from R37 (Creditors/Expenses \- contractors)

        var selfBillingInvoices \= await \_db.SelfBillingInvoices

            .Where(i \=\> i.Year \== year && (\!month.HasValue || i.Month \== month))

            .ToListAsync();

        

        // Get employee salaries (Creditors/Expenses \- employees)

        var employeeSalaries \= await \_db.DriverInvoices

            .Where(i \=\> i.Year \== year && (\!month.HasValue || i.Month \== month))

            .ToListAsync();

        

        // Aggregate by customer (Debtors)

        var debtors \= customerInvoices

            .GroupBy(i \=\> i.ClientId)

            .Select(g \=\> new DebtorSummary

            {

                ClientName \= g.First().Client.Name,

                TotalTurnover \= g.Sum(i \=\> i.TotalInclVAT),

                Outstanding \= g.Sum(i \=\> i.AmountOutstanding)

            })

            .OrderByDescending(d \=\> d.TotalTurnover)

            .ToList();

        

        // Aggregate by contractor (Creditors)

        var creditors \= selfBillingInvoices

            .GroupBy(i \=\> i.ContractorName)

            .Select(g \=\> new CreditorSummary

            {

                ContractorName \= g.Key,

                TotalExpenses \= g.Sum(i \=\> i.TotalInclVAT),

                Outstanding \= g.Sum(i \=\> i.AmountOutstanding)

            })

            .ToList();

        

        return new FinancialSummaryReport

        {

            Period \= month.HasValue ? $"{year}-{month:D2}" : $"{year}",

            TotalRevenue \= debtors.Sum(d \=\> d.TotalTurnover),

            TotalExpenses \= creditors.Sum(c \=\> c.TotalExpenses) \+ employeeSalaries.Sum(s \=\> s.Total),

            Debtors \= debtors,

            Creditors \= creditors

        };

    }

}

## R37: Invoicing System (Customer Invoices \+ Self-Billing) \- FULL ANALYSIS

**⚡ CRITICAL CLARIFICATION**: R37 is the **core invoicing system** that generates TWO types of invoices:

1. **Customer Invoices** (Verkoop/Sales): What customers owe the company  
2. **Self-Billing Invoices** (Inkoop/Purchases): What company owes contractors

### Requirement Summary

Implement **complete invoicing system** with two invoice flows:

**1\. Customer Invoicing (Verkoop-Tarief)**:

- Generate invoices to customers for services rendered  
- Based on completed rides/trips  
- Uses **verkoop-tarief** (selling rate) from contracts  
- Customer pays company

**2\. Self-Billing Invoicing (Inkoop-Tarief)**:

- Generate invoices **on behalf of** ZZP contractors/subcontractors  
- Based on hours worked by contractors  
- Uses **inkoop-tarief** (purchase rate) from contracts  
- Company pays contractor

**Verkoop vs. Inkoop Rates**:

- **Inkoop-Tarief**: What company pays driver/contractor (e.g., €28/hour)  
- **Verkoop-Tarief**: What company charges customer (e.g., €50/hour)  
- **Margin**: Difference is company profit (€50 \- €28 \= €22/hour)

**What is Self-Billing?**:

- You create invoice on behalf of supplier → Supplier approves → You pay  
- Common in Dutch transport/logistics for ZZP drivers and subcontractors

**Legal Requirements** (Dutch tax law):

- Written self-billing agreement required  
- Invoice must state "Self-Billing Invoice" / "Eigen Factuur"  
- Must include supplier's BTW number  
- Supplier must approve each invoice  
- BTW/VAT compliance

### Gap Analysis

**What needs to be added:**

**⚡ KEY CONCEPT \- Dual Rate System**:

- Each contract/trip must store **TWO rates**:  
  - **Inkoop-Tarief** (Purchase/Cost rate): What company pays contractor (e.g., €28/hour)  
  - **Verkoop-Tarief** (Sales/Revenue rate): What company charges customer (e.g., €50/hour)  
  - **Company Margin**: Verkoop \- Inkoop (€50 \- €28 \= €22/hour profit)

**Database Entities**:

1. **CustomerInvoice** (for customer billing \- verkoop):  
     
   - Generated from completed rides/trips  
   - Uses verkoop-tarief from trip/contract  
   - Total \= Revenue for company (Debtors in R36)

   

2. **SelfBillingInvoice** (for contractor payment \- inkoop):  
     
   - Generated from contractor hours worked  
   - Uses inkoop-tarief from contract  
   - Total \= Expense for company (Creditors in R36)

   

3. **SelfBillingAgreement** (legal requirement for self-billing)

**Services**: InvoiceGenerationService (both customer and self-billing), InvoicePdfBuilder

**Frontend**: Invoices page (customer \+ self-billing tabs), generation wizard, contractor approval interface

2. ✅ Contractor approves via email link  
3. ✅ Invoice ready for payment
